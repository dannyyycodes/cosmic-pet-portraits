## [CRITICAL] Client Can Create 100% Discounted Gift-Card Draft Orders

Status: Open
Area: Payments
Files:
- api/cart/checkout.ts:287
- api/cart/checkout.ts:313
- api/cart/checkout.ts:318
- api/_lib/shopifyAdmin.ts:141
Endpoints:
- POST /api/cart/checkout
Safety tag: SAFE

### Impact
An unauthenticated caller can submit a gift-card cart line with any allowed gift-card `variantId` and `appliedDiscountPct: 100`. The API forwards that client-supplied percentage into Shopify draft-order `applied_discount`, preserving the native gift-card variant. If Shopify allows zero-value checkout for the draft, the attacker can mint gift value without payment.

### Evidence
Static PoC body:
```json
{
  "currency": "GBP",
  "items": [{
    "productType": "gift-card",
    "variantId": 64670403527005,
    "properties": {
      "recipient_name": "AUDIT",
      "recipient_email": "audit@example.com"
    },
    "appliedDiscountPct": 100
  }]
}
```
`checkout.ts` trusts `appliedDiscountPct` from the request and accepts values `> 0 && <= 100`; `shopifyAdmin.ts` serializes it directly into `draft_order.line_items[].applied_discount`.

### Expected
Discount authority should be server-side only, tied to a signed promotion/session, and capped by a business rule that cannot be chosen by the browser.

### Actual
The browser chooses the discount percentage for gift cards.

### Root Cause
Gift-card discount metadata was treated as cart state instead of a privileged pricing decision.

### Recommended Fix
Remove `appliedDiscountPct` from accepted public cart input. Resolve gift-card discounts from a server-side allowlist or signed one-time promo token. Reject 100% discounts unless explicitly generated by a trusted admin flow.

### Verification
Direct POST with `appliedDiscountPct: 100` returns 400. Valid server-issued discount token creates the expected capped discount.

## [CRITICAL] Cron Timeout Does Not Cancel In-Flight Gelato Submission

Status: Open
Area: Fulfillment
Files:
- api/cron/gelato-worker.ts:153
- api/cron/gelato-worker.ts:181
- api/cron/gelato-worker.ts:201
- api/_lib/canvasFulfillment.ts:112
- api/_lib/printPipeline.ts:277
Endpoints:
- POST /api/cron/gelato-worker
Safety tag: SAFE

### Impact
A slow row can continue running after the worker's `Promise.race` rejects at 60 seconds. The catch deliberately leaves the row `pending`, so a later cron tick can claim the same row and start another pipeline while the first one may still submit to Gelato. That can create duplicate physical orders and duplicate production cost for one paid Shopify line.

### Evidence
`gelato-worker.ts` claims by incrementing `attempts` but leaves `status='pending'`. It races `runCanvasFulfillmentForRow(claimedRow)` against a 60s timer. The timer rejection does not abort `runPrintPipeline` or its Gelato POST; the catch says not to update status. The next cron query selects `status='pending'` rows again.

### Expected
Once claimed, a row should move to an in-progress/leased state that cannot be claimed by another worker until the lease expires, and external POSTs should be idempotent or guarded by state.

### Actual
Claimed rows remain `pending`, and timeout does not cancel the underlying async fulfillment.

### Root Cause
`Promise.race` is used as a deadline without propagating an `AbortSignal`, and the queue state has no `processing` lease.

### Recommended Fix
Add `processing` or `locked_until` fields. Claim rows with an atomic update from `pending` to `processing`. Pass an `AbortSignal` through AuraSR/Gelato fetches. Before Gelato POST, re-read the row and require it is still owned by the current lease and has no `gelato_order_id`. Use a deterministic Gelato idempotency/reference policy and treat duplicate-reference responses as success.

### Verification
With a mocked slow Gelato call exceeding 60s, a second worker tick must not start a second Gelato POST for the same `(shopify_order_id, shopify_line_item_id)`.

## [CRITICAL] Direct Checkout Allows Paid Canvas Without Print Master

Status: Open
Area: Payments | Fulfillment
Files:
- api/cart/checkout.ts:370
- api/cart/checkout.ts:376
- api/cart/checkout.ts:382
- api/shopify/_lib/extractCanvas.ts:217
- api/shopify/_lib/extractCanvas.ts:227
Endpoints:
- POST /api/cart/checkout
- POST /api/shopify
- POST /api/cron/gelato-worker
Safety tag: SAFE

### Impact
A caller can bypass the frontend approval and print-master preparation, create a Shopify invoice for a physical canvas, and pay for a line that later falls back to source photo or preview artwork. This can ship the wrong image or a low-quality print.

### Evidence
`checkout.ts` requires a print master for `kind === "template"` canvas and for all digital lines, but not for AI physical canvas lines. `extractCanvas.ts` later falls back to `_print_master_url_legacy`, then `Source photo`, then `Preview portrait`.

### Expected
Every paid physical Pawtraits canvas should require a server-created `printMasterPath` or a verified legacy print master, regardless of client `kind`.

### Actual
AI physical lines can omit both `printMasterPath` and `printMasterUrl`.

### Root Cause
The backend trusts the SPA to block cart-add when print-master generation fails.

### Recommended Fix
In checkout, require `printMasterPath` for `productType === "canvas"` and `productType === "framed-canvas"` unless the line is an explicitly supported post-purchase-customisation flow. Remove source/preview fallback for normal paid canvas lines; send missing-master rows to manual review before checkout or before payment.

### Verification
Direct POST for `canvas`/`framed-canvas` without `_print_master_path` returns 400. Shopify webhook tests confirm no normal canvas line can be queued without a private print master.

## [HIGH] SSRF in Cutout, Composite, and Mockup URL Fetches

Status: Open
Area: Security
Files:
- api/portraits.ts:567
- api/portraits.ts:582
- api/portraits.ts:584
- api/portraits.ts:2311
- api/portraits.ts:2352
- api/portraits.ts:2434
Endpoints:
- POST /api/portraits?action=cutout
- POST /api/portraits?action=composite
- POST /api/portraits?action=mockup
Safety tag: SAFE

### Impact
Authenticated users can make `cutout` fetch arbitrary `imageUrl` through the server fallback path, and unauthenticated callers can make `composite` and `mockup` fetch arbitrary `cutoutUrl`/`designUrl`. This exposes the app to SSRF against cloud metadata/internal services and can also burn CPU/storage with attacker-controlled images.

### Evidence
`handleCutout` reads `imageUrl` and passes it to `tryHfRmbg`, which calls `fetch(imageUrl)` without `validateImageUrlOrigin`. `handleComposite` fetches `cutoutUrl` without auth or origin validation. `handleMockup` fetches `designUrl` without auth or origin validation.

### Expected
All server-side fetches of user-supplied URLs should enforce the same HTTPS allowlist used by `generate` and `printMaster_submit`, plus size/content-type limits.

### Actual
These endpoints fetch arbitrary caller-provided URLs.

### Root Cause
SSRF validation was added to the generation path but not consistently applied to template/mockup utilities.

### Recommended Fix
Require auth on `composite` and `mockup`. Apply `validateImageUrlOrigin` before every user-controlled fetch. Prefer storage paths/object IDs over public URLs for internal assets. Enforce response content-type and max byte limits before handing buffers to Sharp.

### Verification
POSTs using `http://169.254.169.254/`, `https://localhost/`, and unapproved external hosts return 400 before any fetch.

## [HIGH] Gelato Webhook Replays Are Skipped After Processing Failure

Status: Open
Area: Webhook | Reliability
Files:
- api/gelato/webhook.ts:168
- api/gelato/webhook.ts:183
- api/gelato/webhook.ts:195
- api/gelato/webhook.ts:207
Endpoints:
- POST /api/gelato/webhook
Safety tag: SAFE

### Impact
If Gelato sends a valid event and `processGelatoEvent` fails after the event is inserted into `gelato_webhook_events`, the handler returns 500, but Gelato's retry will be treated as a replay and skipped. The order status and shipped/delivered touchpoints can be lost permanently.

### Evidence
The handler records the dedupe event before calling `processGelatoEvent`. The catch comments acknowledge that retry will hit the replay branch and 200.

### Expected
Dedupe should only mark an event processed after side effects complete, or should store a processing/failed state that retries can resume.

### Actual
The dedupe row is inserted before the critical status update.

### Root Cause
The event log conflates "seen" with "successfully processed."

### Recommended Fix
Add `status` to `gelato_webhook_events` (`processing`, `processed`, `failed`) and update to `processed` only after side effects succeed. On retry, reprocess `failed` or stale `processing` rows. Alternatively delete the dedupe row in the catch before returning 500, as the Stripe webhook does.

### Verification
Inject a mocked DB failure after dedupe insert. Retry of the same event must re-run status processing and update `print_orders`.

## [HIGH] Print-Master Generation Is an Authenticated but Unmetered Cost Endpoint

Status: Open
Area: API | Security
Files:
- api/portraits.ts:3270
- api/portraits.ts:3320
- api/portraits.ts:3357
- api/portraits.ts:3371
Endpoints:
- POST /api/portraits?action=printMaster_submit
- POST /api/portraits?action=printMaster_status
Safety tag: SAFE

### Impact
Any signed-in account can call `printMaster_submit` repeatedly to trigger high-quality fal generations without consuming credits. This is a paid cost center and can be scripted at account scale.

### Evidence
The endpoint requires a Bearer token, validates image URLs, runs Vision, then submits to fal queue with `quality: 'high'`. The content-policy response explicitly says `creditRefunded: false` because no credit was consumed. No rate-limit or prior-generation/approval ownership check is present.

### Expected
Print-master generation should be tied to an approved generation owned by the user, or consume a metered entitlement, and should be rate-limited.

### Actual
Authentication is the only meaningful throttle.

### Root Cause
The print-master endpoint was designed as a cart-add helper but exposed as a general paid generation API.

### Recommended Fix
Persist print-master jobs server-side with `user_id`, source generation id, and status. Require a successful/approved generation owned by the caller, enforce per-user rate limits, and consume/refund a print-master entitlement or reuse the original generation credit policy.

### Verification
A signed-in account without an approved generation cannot submit print-master jobs. Repeated submissions for the same approved image are deduped or rate-limited.

## [HIGH] Digital Delivery Failures Are Marked Failed and Not Retried

Status: Open
Area: Fulfillment | Reliability
Files:
- api/cron/gelato-worker.ts:125
- api/cron/gelato-worker.ts:128
- api/_lib/canvasFulfillment.ts:275
- api/_lib/canvasFulfillment.ts:278
- api/_lib/digitalFulfillment.ts:153
- api/_lib/digitalFulfillment.ts:169
Endpoints:
- POST /api/cron/gelato-worker
Safety tag: SAFE

### Impact
A transient Resend/API/network failure after payment can leave a digital order in `status='failed'`. The cron worker only selects `status='pending'`, so the customer does not get automatic retry delivery.

### Evidence
Digital fulfillment returns `ok:false` for email non-2xx or thrown errors. The caller updates `print_orders.status` to `failed`. The worker query only drains `status='pending'`.

### Expected
Transient email/download/upload failures should remain retryable up to a cap, then move to manual review with an ops alert.

### Actual
The first transient failure becomes terminal for the automatic worker.

### Root Cause
The digital path inherited a terminal `failed` state instead of a retryable state machine.

### Recommended Fix
For digital rows, keep retryable failures as `pending` with incremented attempts and `last_error`, or add `retry_at`. Move to `manual_review` only after `MAX_ATTEMPTS`. Make Resend email send idempotent with a stable key or store `emailMessageId` before terminal success.

### Verification
Mock Resend returning 500 once then 200. The row should retry and end as submitted with a delivery email.

## [MEDIUM] Checkout Does Not Enforce Required Consent Server-Side

Status: Open
Area: Legal | Payments
Files:
- api/cart/checkout.ts:437
- api/cart/checkout.ts:444
- api/cart/checkout.ts:456
- src/components/portraits/CartConsents.tsx:15
- src/components/portraits/CartDrawer.tsx:120
Endpoints:
- POST /api/cart/checkout
Safety tag: SAFE

### Impact
The UI disables checkout until required consent boxes are ticked, but direct API callers can omit `consent` and still create personalized canvas or immediate-delivery reading draft orders. That weakens the legal cancellation record.

### Evidence
`checkout.ts` only writes consent metafields if timestamps are present. It does not reject a physical personalized item or Soul Reading when the corresponding timestamp is absent.

### Expected
The server should reject checkout bodies missing required consent for cart contents.

### Actual
Consent is optional at the API boundary.

### Root Cause
Legal gating is implemented only in client state.

### Recommended Fix
Derive required consent server-side from normalized line items. Reject physical personalized items without `canvasPersonalisedAt` and Soul Reading lines without `readingImmediateAt`. Validate timestamps are plausible ISO datetimes.

### Verification
Direct checkout POST with canvas items and no consent returns 400. UI checkout with consent still succeeds.

## [MEDIUM] Supabase PaymentIntent Function Creates Placeholder Reports Before Payment Without Rate Limit

Status: Open
Area: API | Reliability
Files:
- supabase/functions/create-payment-intent/index.ts:30
- supabase/functions/create-payment-intent/index.ts:94
- supabase/functions/create-payment-intent/index.ts:121
Endpoints:
- POST /functions/v1/create-payment-intent
Safety tag: SAFE

### Impact
Unauthenticated callers can create up to 10 `pet_reports` rows per request before any payment is completed. Without a rate limit, this can spam production data, create abandoned pending reports, and increase downstream cleanup/ops burden.

### Evidence
The function validates only request shape, creates placeholder reports in a loop, then creates a Stripe PaymentIntent. It does not call the shared `rate-limit.ts` helper and does not require a user/session.

### Expected
Unauthenticated pre-payment mutation should be rate-limited and/or delayed until payment confirmation where possible.

### Actual
DB rows are created before payment on an open endpoint.

### Root Cause
Express checkout needs a PaymentIntent client secret, but placeholder persistence is not protected by rate limiting.

### Recommended Fix
Apply IP/device/email rate limiting. Store pre-payment checkout attempts in a short-lived table or create reports only after `payment_intent.succeeded`/verification. Add cleanup for stale `pending@checkout.temp` rows.

### Verification
Repeated unauthenticated calls from the same IP exceed a low threshold and return 429 without new report rows.

## [LOW] Shopify Checkout Return URL Trusts Request Origin

Status: Open
Area: Payments | Security
Files:
- api/cart/checkout.ts:486
- api/cart/checkout.ts:488
- api/cart/checkout.ts:490
Endpoints:
- POST /api/cart/checkout
Safety tag: SAFE

### Impact
A cross-origin caller can influence the Shopify draft order `return_url` via the `Origin` header. Payment still happens at Shopify, but after payment the customer could be redirected to an attacker-controlled domain if Shopify accepts that URL.

### Evidence
`checkout.ts` computes `origin` from `req.headers.origin` before falling back to host/default and passes `${origin}/pawtraits/thanks` into `createDraftOrder`.

### Expected
Post-payment return URLs should be pinned to the canonical site origin.

### Actual
The API uses the request's Origin header.

### Root Cause
Convenience for local/dev origin handling was left in the production checkout path.

### Recommended Fix
Use `PUBLIC_SITE_URL` or a hardcoded allowlist. If the request origin is not in the allowlist, fall back to `https://www.littlesouls.app`.

### Verification
POST with `Origin: https://evil.example` creates a draft order whose return URL remains `https://www.littlesouls.app/pawtraits/thanks`.
