# Production-readiness audit — 2026-05-11

## Summary
**Conditional ship.** Core money-loss invariants hold (every credit-consume on a failure path is paired with `safeRefund`; printMaster 502 hard-stops cart-add; Stripe webhook is idempotent). One config-dependent P0 blocks every default cart add at default Vercel envs. Confidence medium-high once that env is verified.

## P0 issues

**1. Print master gate blocks the default cart add.** `api/portraits.ts:2866` defaults `LARGE_FORMAT_LONG_EDGE_IN=16`, but `StudioFlow.tsx:623` defaults `SAFE_LONG_EDGE_IN=36` and the default `sizeKey` is `"16x20"` (long edge 20). Without `LARGE_FORMAT_ENABLED=true` set on Vercel, every default Add-to-Cart hits `handlePrintMaster` → `sizeForSku().blocked === true` → 503 `size_temporarily_unavailable` → `StudioFlow.handleAdd:1118-1126` shows "Couldn't prepare print master" toast, cart never adds. Customer never charged (good), but conversions fall to zero. **Fix:** set `LARGE_FORMAT_ENABLED=true` (or `LARGE_FORMAT_LONG_EDGE_IN=36`) on Vercel before launch, and surface 503-specific copy in `handleAdd` so future regressions don't silently kill checkout.

## P1 issues

**1. Room mockup silently absent for 16x24 and 24x24.** `PRINTFUL_VARIANT_MAP` (`api/portraits.ts:77-87`) lacks these two sizes. Backend correctly returns `{ error: 'no_mockup_for_size_*' }` and frontend's `PRINTFUL_MOCKUP_SUPPORTED_SIZES` (`StudioFlow.tsx:663-666`) excludes them so the loading skeleton is suppressed. But customers picking those sizes see *no* in-room preview while every other size does — degraded UX with no explanation. Add a one-line caption ("Preview unavailable for this size — final canvas: 16×24″").

**2. Polling effect MAX_POLLS abandon strands credit + portrait.** `StudioFlow.tsx:997-1002`: after 5 min the client gives up with toast "We'll keep your credit safe — please refresh and try again." If the fal job *does* complete server-side later, the row's status moves to `success` but the customer has no UI path to retrieve it. Credit was consumed; portrait exists but is invisible to the customer.

**3. printMaster is synchronous (no fal queue).** `StudioFlow.tsx:1090-1093` flags this in a TODO. `handlePrintMaster` calls `generateVariant` directly (`api/portraits.ts:3149-3166`) at quality `'high'` (30-60s typical). 120s client timeout vs 300s server `maxDuration` gives margin, but a slow fal day will trip the AbortError → toast → no charge → no order. Ship-blocking only if fal latency spikes; recommend queue refactor before scaling.

**4. AuraSR/Gelato bounded retries depend on Telegram alert wiring.** `gelato-worker.ts:189-201` retries 3× then flips `manual_review` and calls `recordHighSeverityFailure` (`canvasFulfillment.ts:189-197`). Customer paid — alert *should* fire, but only if the Telegram side is live. **Verify before launch.**

## P2 issues

- `handleAdd:1060` debounce on `preparingPrintMaster` is best-effort — a fast double-tap before React re-renders can theoretically fire two requests; low real-world risk.
- `StudioFlow.tsx:776-778` resets `approved` on every prompt change including whitespace edits — mildly noisy.
- Stripe webhook's "delete dedup row on handler error" (`api/stripe/webhook.ts:131-137`) is correct for retry semantics — well thought-out, not an issue.

## Things that look correct

**Credit-refund invariant holds end-to-end.** Every `consume_credits` in `handleGenerate` is paired with `safeRefund` on every failure path: log-insert-failed (1976), fal-balance-exhausted (1988), content-policy (1998), submit-failed (2013), exception (2057). `handleGenerationStatus` refunds on missing_fal_urls (2152), fal-job-failed (2176), completed_but_no_url (2208). `safeRefund` itself is exception-safe and writes to `credit_refund_failures` for sweeper recovery (1520-1557). **`handleGenerationStatus` is fully idempotent** — success/failed terminal states return cached results without touching fal (2126-2143). Stripe webhook is idempotent via `portraits_stripe_events` PK + per-invoice `metadata->>invoice_id` belt-and-braces (199-213). **Shopify→Gelato carries sizeKey+frameColor cleanly:** variant_id reverse-mapped via `VARIANT_ID_TO_CANVAS` (`extractCanvas.ts:60-76`), persisted into `print_orders.size_key/frame_color`, read back by `runCanvasFulfillmentForRow` (`canvasFulfillment.ts:243-245`), and fed to `gelatoProductUid()` to build the exact catalog UID — no risk of SKU mismatch between checkout and Gelato. Webhook 500s on DB failure to force Shopify retry. Cron stuck-row sweeper (`gelato-worker.ts:65-116`) catches lambda-killed pipelines. localStorage state persistence TTLs at 24h, validates schema before restore, and defends `sizeKey` restore against stale entries via the `SAFE_LONG_EDGE_IN` length check. `pendingJobId` resumes polling on tab refresh — no orphaned generations on refresh.
